## 2. AI 전략 (AI Strategy)

### 2.1 3계층 AI 아키텍처

```
┌─────────────────────────────────────────────────────────┐
│  Layer 3: Conversational AI (대화형)                     │
│  • AI Agent (LLM)                                        │
│  • 자연어 질의응답, 의사결정 추천                          │
│  • OpenAI GPT-4o / Claude 3.5 / 사내 sLM 하이브리드      │
├─────────────────────────────────────────────────────────┤
│  Layer 2: Specialized ML Models (전문 모델)              │
│  • 예지보전 (시계열 이상탐지, RUL 예측)                   │
│  • 품질 이상탐지 (SPC + AutoEncoder)                     │
│  • 자체 학습한 XGBoost / LSTM / Isolation Forest        │
├─────────────────────────────────────────────────────────┤
│  Layer 1: Rule-based + Statistical (룰/통계)             │
│  • SPC 관리도 (X-bar R, p-chart)                         │
│  • 임계값 기반 알람                                       │
│  • 데이터 부족 초기 6개월 주력                            │
└─────────────────────────────────────────────────────────┘
```

### 2.2 점진적 AI 적용 전략

| 시기 | 주력 Layer | 사유 |
|------|-----------|------|
| Phase 2 (2026.09-11) | Layer 1 (룰/통계) | 학습 데이터 부족, 신뢰성 우선 |
| Phase 3 초기 (2026.12-01) | Layer 1 + Layer 2 (학습 시작) | 3~6개월 데이터 축적 후 ML 학습 |
| Phase 3 후기 (2027.02) | Layer 2 + Layer 3 (Agent 도입) | 모델 안정화, LLM 결합 |
| 운영 (2027.05~) | 3 Layer 통합 | 자율 운영 |

## 3. 예지보전 (Predictive Maintenance, PdM)

### 3.1 적용 대상 설비

| 설비 그룹 | 모니터링 항목 | 주력 분석 |
|-----------|--------------|-----------|
| 프레스 (PRESS-01~08) | 진동(X/Y/Z), 메인모터 전류, 슬라이드 온도, 유압, 소음 | 베어링 결함, 클러치 마모, 다이쿠션 이상 |
| 코일 피더 | 토크, 장력 | 송재 불량 예측 |
| 컴프레서 | 전류, 압력 | 효율 저하 감지 |
| 금형 | 누적 타수, 클리어런스 변화 (간접) | 수명 예측 |

### 3.2 예지보전 기능 명세

| 기능 ID | 기능명 | 우선순위 | 설명 |
|---------|--------|---------|------|
| PDM-01 | 실시간 이상 감지 (룰) | M | 진동/온도/전류 임계값 초과 시 알람 |
| PDM-02 | 통계 기반 이상 감지 | M | 평소 대비 ±3σ 이탈 감지 (SPC) |
| PDM-03 | 시계열 이상탐지 (ML) | S | LSTM-AutoEncoder 재구성 오차 |
| PDM-04 | 고장 확률 예측 | S | 향후 7일/30일 고장 확률 (%) 추정 |
| PDM-05 | 잔여수명 예측 (RUL) | C | 베어링/클러치 잔여수명 (시간) |
| PDM-06 | 정비 권고 생성 | S | AI 알람과 함께 권고 정비 작업 출력 |
| PDM-07 | 정비 이력 학습 피드백 | S | "실제 정비 결과 vs 예측" 자동 학습 |
| PDM-08 | 부품 교체 시기 추천 | S | 베어링/씰/유압호스 교체 시기 |

### 3.3 모델 설계

#### 3.3.1 진동 기반 베어링 결함 감지
- **입력**: 진동 가속도 (8kHz 샘플링, 1초 윈도우)
- **전처리**: FFT → 주파수 도메인 (BPFO, BPFI, BSF, FTF 특징점)
- **모델**: Isolation Forest (이상점 점수) + XGBoost (정상/주의/위험 분류)
- **목표 성능**: F1 ≥ 0.85, FPR ≤ 5%
- **재학습 주기**: 분기 1회 (정비 데이터 축적 후)

#### 3.3.2 전류 기반 모터 이상
- **입력**: 메인모터 3상 전류 (1초 RMS)
- **모델**: LSTM-AutoEncoder (24시간 시퀀스 재구성)
- **출력**: 재구성 오차 → 임계값 초과 시 알람

#### 3.3.3 잔여수명 (RUL) 예측 - Phase 4
- **데이터**: 정비 이력 + 누적 타수 + 진동/온도 추세
- **모델**: Survival Analysis (Cox regression) 또는 GradientBoosting
- **출력**: 95% 신뢰구간 잔여 운전시간

### 3.4 사용자 스토리

**Story PDM-US-01**: 보전팀장
> "PRESS-05에서 어제 새벽 3시에 진동이 평소 대비 2배로 튀었다고 휴대폰에 알림이 왔다.
> 출근해서 보니 AI가 '베어링 BPFO 특성주파수 출현, 7일 내 정비 권고'라고 메시지를 줬다.
> 이전에 비슷한 사례가 있었는지 클릭 한 번에 보고 싶다."

**Story PDM-US-02**: 공장장
> "이번 달 예지보전으로 미리 잡은 고장 건수와, 그로 인해 절감된 다운타임 시간을 월말 보고에 쓰고 싶다."

### 3.5 수용 기준
- 룰 기반 알람: 임계값 초과 → 1초 이내 알람
- ML 모델: 일일 1회 야간 배치 + 5분 간격 추론
- 모델 정확도 모니터링: 매주 자동 리포트 (Precision/Recall/F1)
- 오탐 발생 시 작업자 피드백 → 재학습 데이터에 반영

## 4. 품질 이상탐지 (Quality Anomaly Detection)

### 4.1 기능 명세

| 기능 ID | 기능명 | 우선순위 | 설명 |
|---------|--------|---------|------|
| QAD-01 | SPC 관리도 자동 생성 | M | X-bar R, p-chart (Western Electric 룰) |
| QAD-02 | 공정 능력 지수 (Cp/Cpk) | M | 부품×공정별 자동 계산 |
| QAD-03 | 다변량 이상 탐지 | S | 진동×하중×온도 결합 이상 탐지 (PCA) |
| QAD-04 | 불량 원인 추정 | S | 불량 발생 시 직전 1시간 데이터 분석으로 원인 후보 제시 |
| QAD-05 | 4M 변경 영향도 분석 | S | 작업자/금형/자재 변경 후 품질 변화 자동 비교 |
| QAD-06 | 입고 자재-불량 상관관계 | S | "특정 공급처 LOT일 때 불량률 상승" 자동 발견 |
| QAD-07 | 비전 외관검사 | C | 카메라 + 딥러닝 (CNN) — 2차 사업 후보 |

### 4.2 모델 설계

#### 4.2.1 다변량 이상탐지 (PRS 핵심)
- **입력**: 프레스 1샷당 데이터(타발력 곡선 200포인트, 진동, 슬라이드 위치)
- **모델**: PCA + Mahalanobis Distance, Hotelling's T²
- **출력**: 정상 / 주의 / 이상 (3단계)
- **활용**: 이상 샷의 부품을 자동으로 격리 및 추적 마킹

#### 4.2.2 불량 원인 추정 (인과 추론)
- 룰 기반 우선: "불량 발생 직전 1시간 내 4M 변경이 있었는가?"
- 상관분석: 진동/온도/전류 중 평소 패턴 대비 가장 많이 벗어난 변수 Top-3 제시
- 한계: 인과 ≠ 상관 → "추정"으로 명시, 작업자가 최종 판단

## 5. AI Agent (LLM 기반 통합 질의)

> 사업계획서 p41 "AI Agent 관리" 모듈의 구체화. 모든 모듈에 횡단적으로 통합되는 인터페이스.

### 5.1 핵심 컨셉

```
┌────────────────────────────────────────────────┐
│  사용자: "오늘 PRESS-03 불량률 알려줘"          │
│             ↓                                  │
│  [의도 분석 (Intent Parser)]                    │
│             ↓                                  │
│  [SQL/API 호출 생성 (Tool Use)]                 │
│             ↓                                  │
│  [데이터 조회 → 결과 요약 → 자연어 답변]         │
│             ↓                                  │
│  AI: "오늘 06:00~14:00 PRESS-03 생산량 3,420개, │
│       불량 12개로 불량률 0.35%입니다.            │
│       어제 동시간 0.18%보다 0.17%p 높습니다."   │
└────────────────────────────────────────────────┘
```

### 5.2 AI Agent 기능 명세

| 기능 ID | 기능명 | 우선순위 | 설명 |
|---------|--------|---------|------|
| AGT-01 | 통합 자연어 질의 | M | 모든 모듈 데이터에 대한 자연어 Q&A |
| AGT-02 | 생산 현황 요약 | M | "오늘 생산 현황 요약해줘" |
| AGT-03 | 품질 분석 답변 | M | "이번 주 가장 불량이 많은 공정은?" |
| AGT-04 | LOT 추적 자동화 | M | "LOT-XXX 추적해줘" → 자동 보고서 |
| AGT-05 | 의사결정 추천 | S | "PRESS-03 정비 지금 해야 할까?" |
| AGT-06 | 알림 및 추천 | S | 능동적 알림 ("3시간 후 납기 임박 건 있음") |
| AGT-07 | 음성 입출력 | S | 현장 작업자 헤드셋 연동 |
| AGT-08 | 사용자 질문이력 | M | 모든 대화 저장, 검색 가능 |
| AGT-09 | 답변 신뢰도 표시 | M | "이 답변은 데이터 X건 기반"으로 근거 명시 |
| AGT-10 | 권한 기반 답변 | M | RBAC: 오퍼레이터는 자기 라인만, 관리자는 전체 |

### 5.3 기술 스택

| 구분 | 선택 | 사유 |
|------|------|------|
| 메인 LLM | Claude 3.5 Sonnet / GPT-4o | 한국어 우수, Tool Use 강력 |
| 보조 sLM | Llama 3 / Qwen2 (사내 GPU) | 비용 절감, 보안 |
| 검색 | RAG (벡터 DB) | 매뉴얼/SOP 검색용 |
| Tool Use | Function Calling | DB 조회, API 호출 |
| 캐싱 | Redis | 동일 질문 비용 절감 |

### 5.4 RAG 데이터 소스
- 작업표준서 (SOP)
- 4M 변경 이력
- 과거 정비 기록
- 모회사 품질 요구사항 문서
- 사내 매뉴얼

### 5.5 사용자 스토리

**Story AGT-US-01**: 작업반장 (모바일)
> "퇴근 직전에 휴대폰에 대고 '오늘 우리 라인 요약해줘' 하면, 생산량, 불량, 가동률, 특이사항을 3줄로 말해주면 좋겠다."

**Story AGT-US-02**: 품질 엔지니어
> "'지난 주에 갑자기 불량률이 올라간 원인 분석해줘' 라고 물으면, AI가 SPC 차트 + 4M 변경 이력 + 자재 LOT을 조합해서 후보 원인 3개를 제시해주면 좋겠다."

**Story AGT-US-03**: 영업/CS
> "현대모비스에서 클레임이 왔다. LOT 번호만 입력하면 AI가 즉시 해당 LOT의 추적 보고서(PDF)를 만들어 메일로 보내주면 좋겠다."

### 5.6 수용 기준
- 평균 응답 시간 ≤ 5초
- 환각(Hallucination) 방지: 데이터 근거 없는 답변 금지, "확인 필요" 답변 허용
- 모든 답변에 출처(어느 테이블/문서) 표시
- 권한 외 데이터 접근 차단 (사번별 RBAC)
- 비용 관리: 사용자당 일 100회 제한 (관리자 설정)

### 5.7 위험요소

| 위험 | 대응 |
|------|------|
| LLM 환각 (잘못된 답변) | Tool Use 강제, RAG 출처 노출, 신뢰도 표시 |
| 토큰 비용 폭증 | 캐싱, sLM 라우팅, 사용량 제한 |
| 민감 데이터 외부 유출 | 사내 sLM 라우팅 (모회사 가격 정보 등) |
| 자연어 SQL 인젝션 | LLM이 직접 SQL 생성 금지, 사전정의 Tool만 호출 |

## 8. AI 운영 거버넌스

### 8.1 모니터링
- 모델 정확도 주간 리포트 (자동)
- Data Drift 감지 (입력 분포 변화)
- 인간 피드백 수집 (작업자가 "맞음/틀림" 평가)

### 8.2 윤리 및 안전
- AI 알람만으로 자동 정비/생산정지 금지 → 반드시 사람 승인
- 작업자 평가/인사에 AI 데이터 사용 금지 명시
- 모회사 가격/원가 데이터는 외부 LLM 호출 차단
